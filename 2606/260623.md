# Spring Security + JWT 인증 흐름

---

## 1. 로그인

### 요청

```http
POST /api/v1/auth/login
```

```json
{
  "email": "test@test.com",
  "password": "1234"
}
```

### 처리 과정

1. `AuthController` → `AuthService` 호출
2. `AuthenticationManager.authenticate()` 호출
3. `UserDetailsService.loadUserByUsername()` → DB에서 사용자 조회
4. `PasswordEncoder.matches()` → 비밀번호 검증
5. JWT Access Token 생성
6. JWT Refresh Token 생성
7. Refresh Token 저장 (DB 또는 Redis)
8. Access Token + Refresh Token 반환

> **⚠️ 주의:** 비밀번호 검증을 `AuthService`에서 직접 처리하면 Spring Security 인증 메커니즘을 우회하게 됩니다.
> `AuthenticationManager`를 통해 처리하는 것이 Spring Security의 올바른 사용 방법입니다.

### 응답

- Access Token → Response Body
- Refresh Token → HttpOnly Cookie

### 흐름도

```
Client
   │
   │ POST /api/v1/auth/login
   ▼
AuthController
   ▼
AuthService
   ▼
AuthenticationManager.authenticate()
   ▼
UserDetailsService.loadUserByUsername()   ← DB 사용자 조회
   ▼
PasswordEncoder.matches()                 ← 비밀번호 검증
   ▼
JWT 생성
   ▼
AccessToken → Response Body
RefreshToken → HttpOnly Cookie
```

## 2. API 요청 (인증)

### 요청

```http
GET /api/v1/recipes
Authorization: Bearer accessToken
```

### 처리 과정

1. `JwtAuthenticationFilter` 실행
2. `Authorization` 헤더 확인
3. Access Token 추출
4. JWT 유효성 검증
    - 서명 검증
    - 만료 여부 확인
5. 사용자 정보 처리 (아래 두 방식 중 선택)
6. `SecurityContext`에 `UsernamePasswordAuthenticationToken` 저장
7. Controller 실행

#### 사용자 정보 처리 방식 비교

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| JWT Claims만 사용 | DB 조회 없이 토큰에서 사용자 정보 추출 | 빠름, DB 부하 없음 | 탈취된 토큰 즉시 차단 불가 |
| DB 조회 방식 | `UserDetailsService`로 DB에서 사용자 조회 | 계정 상태 실시간 반영 가능 | DB 요청 증가 |

> 어느 방식을 사용하더라도 `SecurityContext`에 `UsernamePasswordAuthenticationToken`을 저장하는 것은 동일합니다.

### 흐름도

```
Client
   │
   │ Authorization: Bearer AccessToken
   ▼
JwtAuthenticationFilter
   ▼
Token 유효성 검증 (서명, 만료)
   ▼
사용자 정보 추출
(JWT Claims 또는 DB 조회)
   ▼
SecurityContext에 Authentication 저장
   ▼
Controller 실행
```

## 3. Access Token 만료

### 상황

```
AccessToken 만료
RefreshToken 유효
```

API 호출 시 `401 Unauthorized` 발생

### 클라이언트 처리 흐름

```
API 요청
   ▼
401 Unauthorized 수신
   ▼
POST /api/v1/auth/refresh 자동 호출     ← axios interceptor 등으로 처리
   ▼
새 Access Token 수신
   ▼
원래 요청 재시도
```

> **참고:** 401 이후의 재시도 로직은 프론트엔드(axios interceptor 등)에서 처리합니다.
> 백엔드는 `/auth/refresh` 엔드포인트만 제공하면 됩니다.

## 4. 토큰 재발급

### 요청

```http
POST /api/v1/auth/refresh
```

### 처리 과정

1. HttpOnly Cookie에서 `refreshToken` 조회

```java
@CookieValue(name = "refreshToken") String refreshToken
```

2. `AuthService.refreshToken()` 호출
3. Refresh Token 검증
    - 서명 검증
    - 만료 여부 확인
    - DB/Redis 저장 여부 확인
4. 새로운 Access Token 생성
5. 새로운 Refresh Token 생성
6. 기존 Refresh Token 폐기 (Refresh Token Rotation)
7. 새로운 Refresh Token 저장
8. 응답 반환

### 응답

- Access Token → Response Body
- Refresh Token → HttpOnly Cookie

### 흐름도

```
Client
   │
   │ POST /api/v1/auth/refresh (Cookie: refreshToken)
   ▼
AuthController
   ▼
쿠키에서 RefreshToken 조회
   ▼
RefreshToken 검증 (서명, 만료, DB/Redis 저장 여부)
   ▼
기존 RefreshToken 폐기
   ▼
새 AccessToken + RefreshToken 발급
   ▼
AccessToken → Body
RefreshToken → HttpOnly Cookie
```

## 5. 로그아웃

### 요청

```http
POST /api/v1/auth/logout
Authorization: Bearer accessToken
```

### 처리 과정

1. Cookie에서 Refresh Token 조회
2. DB/Redis에서 Refresh Token 삭제
3. **Access Token 블랙리스트 처리** (Redis 등에 저장)
4. 만료 쿠키 생성

```java
.maxAge(0)
```

5. 브라우저 쿠키 삭제

> **⚠️ Access Token 처리 주의:**
> Refresh Token만 삭제하면 Access Token이 만료될 때까지 여전히 유효합니다.
> 이를 방지하기 위한 방법은 두 가지입니다.
>
> | 방법 | 설명 |
> |------|------|
> | 짧은 Access Token 만료 시간 | 예) 15분 이내로 설정해 위험 노출 시간 최소화 |
> | Redis 블랙리스트 | 로그아웃된 토큰을 Redis에 저장, JWT 필터에서 확인 |

### 흐름도

```
Client
   │
   │ POST /api/v1/auth/logout
   ▼
AuthController
   ▼
RefreshToken 삭제 (DB/Redis)
   ▼
AccessToken 블랙리스트 등록 (Redis)
   ▼
쿠키 만료 처리 (maxAge=0)
   ▼
로그아웃 완료
```
