## 데이터베이스 논리 모델링(ERD) 설계

레시피 추천 및 냉장고 관리 서비스 구현을 위한 데이터베이스 논리 모델링을 수행하였다. 서비스 요구사항을 분석하여 회원, 냉장고, 식재료, 레시피, 댓글, 좋아요, 장바구니, 장보기 이력 등 주요 도메인을 식별하고 엔티티 및 관계를 설계하였다.

### 주요 테이블 설계

| 테이블명 | 주요 역할 |
|----------|----------|
| 회원(Member) | 사용자 정보 관리 |
| 냉장고(Refrigerator) | 사용자별 냉장고 관리 |
| 냉장고_재료(Refrigerator_Ingredient) | 냉장고 내 식재료 관리 |
| 식재료(Ingredient) | 식재료 정보 관리 |
| 식재료_카테고리(Ingredient_Category) | 식재료 분류 관리 |
| 레시피(Recipe) | 레시피 기본 정보 관리 |
| 레시피_순서(Recipe_Step) | 조리 순서 관리 |
| 레시피_재료(Recipe_Ingredient) | 레시피와 식재료 관계 관리 |
| 댓글(Comment) | 레시피 댓글 관리 |
| 좋아요(Recipe_Like) | 레시피 좋아요 관리 |
| 장바구니(Cart) | 장보기 목록 관리 |
| 장보기_이력(Purchase_History) | 구매 이력 관리 |

### 엔티티 관계 설계

#### 1:N 관계

- 회원 → 냉장고
- 냉장고 → 냉장고_재료
- 레시피 → 레시피_순서
- 레시피 → 댓글
- 회원 → 댓글

#### N:M 관계 해소

레시피와 식재료는 다대다(N:M) 관계가 발생하므로 중간 테이블을 이용하여 관계를 해소하였다.

```text
 레시피
   1
   │
   │
   N
레시피_재료
   N
   │
   │
   1
 식재료
```

예시 데이터

| recipe_id | ingredient_id |
|------------|--------------|
| 1 | 1 (김치) |
| 1 | 2 (돼지고기) |
| 1 | 3 (두부) |
| 2 | 3 (두부) |
| 2 | 4 (된장) |

이를 통해 하나의 레시피에 여러 식재료를 등록할 수 있으며, 하나의 식재료를 여러 레시피에서 재사용할 수 있도록 설계하였다.

### 데이터베이스 설계 시 고려사항

- 모든 테이블에 PK 설정
- 테이블 간 참조 무결성을 위한 FK 설정
- 필수 데이터에 NOT NULL 제약조건 적용
- 카테고리 테이블 분리를 통한 데이터 중복 최소화
- 정규화를 통한 데이터 일관성 확보
- 사용자별 냉장고 재고 관리가 가능하도록 구조 설계
- 레시피 추천 기능 구현을 고려한 식재료 연관관계 설계
- 댓글 및 좋아요 기능 확장을 고려한 사용자 활동 데이터 관리
- 향후 JPA 연관관계 매핑을 고려한 테이블 구조 설계
