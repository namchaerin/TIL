# Data Modeling
## 개요
데이터 모델링은 현실 세계의 데이터를 데이터베이스에 저장하기 위해 구조화하는 과정이다.  
데이터를 체계적으로 관리할 수 있도록 테이블, 컬럼, 관계 등을 설계하는 작업이다.

데이터 모델링을 수행하면 다음과 같은 장점이 있다.

- 데이터 구조를 한눈에 파악할 수 있다.
- 테이블 간 관계를 쉽게 이해할 수 있다.
- 데이터 중복을 줄일 수 있다.
- 유지보수성과 확장성을 높일 수 있다.

관계형 데이터베이스 설계 시에는 다음 요소들을 고려해야 한다.

- 어떤 기준으로 테이블을 분리할 것인지 정의
- 컬럼의 의미와 역할 정의
- 테이블 간 관계 설정
- 데이터 중복 최소화

또한 소프트웨어 개발 절차와 데이터 모델링의 관계도 학습하였다.

일반적인 개발 절차는 다음과 같다.

```text
요구사항 분석 → 설계 → 구현 → 테스트 → 유지보수
```

정보 시스템은 데이터를 입력받아 가공하고 필요한 정보를 제공하는 시스템이며, 데이터 모델링은 이러한 정보 시스템 구축 과정에서 중요한 역할을 수행한다.

데이터 모델링은 크게 다음과 같이 구분된다.

- 데이터 관점 모델링
- 프로세스 모델링
- 상관 모델링

데이터 관점 모델링에서는 ERD(Entity Relationship Diagram)를 설계하고 데이터베이스 구조를 정의한다.

프로세스 모델링은 업무 흐름과 처리 절차를 표현하는 과정이다.

상관 모델링은 데이터와 업무 간 영향을 분석하는 과정이다.

---

ERD 표기법으로는 다음과 같은 방식이 존재한다.

- 피터첸 표기법
- IE 표기법
- 바커 표기법

IE 표기법은 범용적으로 많이 사용되며, 바커 표기법은 논리 모델링을 보다 구체적으로 표현할 수 있다는 특징이 있다.

관계형 모델(Relational Model)에서는 무결성 개념이 중요하다.

무결성은 데이터가 정확하고 완전한 상태를 의미한다.

정합성은 데이터 간 모순이 없는 상태를 의미하며, 무결성과는 차이가 있다.

무결성의 종류는 다음과 같다.

| 무결성 종류 | 설명 |
| --- | --- |
| 엔터티 무결성 | 기본키는 NULL이 될 수 없고 고유해야 함 |
| 참조 무결성 | 외래키는 참조 대상의 기본키 값이어야 함 |
| 도메인 무결성 | 컬럼은 정의된 범위의 값만 허용 |
| 업무 무결성 | 업무 규칙에 맞는 데이터 유지 |

## 데이터 모델링 단계 정리

### 1. 개념 모델(Conceptual Model)

업무 중심으로 데이터를 큰 흐름에서 표현하는 단계이다.

특징
- 사용자 관점 중심
- 핵심 엔터티와 관계 표현
- 세부 컬럼이나 데이터 타입은 표현하지 않음
- 업무 구조 파악 목적

예시
- 회원
- 주문
- 상품

같은 주요 객체와 관계만 표현한다.

주요 목적
- 업무 이해
- 요구사항 분석
- 전체 구조 설계

---

### 2. 논리 모델(Logical Model)

개념 모델을 기반으로 실제 데이터 구조를 구체화하는 단계이다.

특징
- 엔터티 상세화
- 속성(Attribute) 정의
- 기본키(PK), 외래키(FK) 정의
- 정규화 수행
- DBMS와는 독립적

예시
- 회원번호
- 회원명
- 주문번호
- 상품번호

등 실제 컬럼 수준까지 설계한다.

주요 목적
- 데이터 구조 구체화
- 데이터 중복 제거
- 관계 명확화

---

### 3. 물리 모델(Physical Model)

논리 모델을 실제 DBMS 환경에 맞게 구현하는 단계이다.

특징
- 실제 테이블 생성
- 데이터 타입 정의
- 인덱스 생성
- 성능 고려
- DBMS 종속적

예시
```sql
CREATE TABLE MEMBER (
    MEMBER_ID INT PRIMARY KEY,
    MEMBER_NAME VARCHAR(100)
);
```

주요 목적
- 실제 데이터베이스 구축
- 성능 최적화
- 저장 구조 설계

---

## 모델링 단계 흐름

```text
개념 모델
    ↓
논리 모델
    ↓
물리 모델
```

---

## 단계별 비교

| 구분 | 개념 모델 | 논리 모델 | 물리 모델 |
| --- | --- | --- | --- |
| 관점 | 업무 중심 | 데이터 구조 중심 | DB 구현 중심 |
| 목적 | 요구사항 분석 | 구조 설계 | 실제 구축 |
| 상세 수준 | 낮음 | 중간 | 높음 |
| DBMS 의존성 | 없음 | 없음 | 있음 |
| 주요 내용 | 엔터티, 관계 | 컬럼, PK/FK | 테이블, 타입, 인덱스 |
