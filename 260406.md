# Git 명령어 활용 및 협업 워크플로우

---
## 학습내용
### 1. Git 기본 명령어 (로컬 및 원격 저장소)

- add: 작업 디렉토리의 변경 내용을 스테이징 영역(Staging Area)에 추가.

- commit: 스테이징 영역의 파일을 로컬 저장소에 기록. (의미 있는 단위의 작업 완료)

- push: 로컬 저장소의 커밋 내역을 원격 저장소(GitHub 등)로 업로드.

- pull: 원격 저장소의 최신 변경 사항을 가져와 로컬 브랜치와 병합.

- merge: 서로 다른 브랜치의 변경 사항을 하나로 합침.

### 2. 작업 취소 및 복구 (Revert vs Reset)

- revert: 특정 커밋을 취소하는 새로운 커밋을 생성함. 기존 히스토리를 유지하므로 협업 시 안전함.

- reset: 헤드(HEAD) 위치를 이전 커밋으로 되돌림. 이미 Push 된 커밋에 사용 시 동료의 작업과 꼬일 수 있어 주의가 필요함.

### 3. 브랜치 전략 (Branching)

- 브랜치 생성: git branch feature/greeting - 기능별 독립적인 개발 공간 확보.

- 브랜치 이동: git switch feature/greeting - 작업 환경 전환. (기존 checkout 대신 권장됨)

### 4. 협업 프로세스 (Open Source & Team Project)

- conflict(충돌): 같은 파일의 동일한 라인을 동시에 수정했을 때 발생하며, 개발자가 직접 코드를 비교하여 수정 후 다시 커밋해야 함.

- fork: 타인의 Repository를 나의 원격 저장소로 복제.

- Pull Request(PR): 내가 작업한 내용을 원본 저장소 관리자에게 병합해달라고 요청하는 핵심 협업 도구.

- upstream: Fork 해온 원래의 원본 저장소를 지칭. git remote add upstream [URL]을 통해 원본의 업데이트 사항을 지속적으로 추적할 수 있음.

---
## 느낀 내용
revert를 통해 실수를 안전하게 복구하는 법과 upstream을 활용해 원본 저장소와 동기화하는 법을 배워 실무적인 형상 관리 흐름을 이해할 수 있었다.

---
## 활용 방안
- 팀 프로젝트 적용
- 오픈소스 기여: Fork와 Upstream 관리 기능을 활용하여 외부 라이브러리의 버그 수정이나 기능 개선에 참여해 볼 계획이다.
